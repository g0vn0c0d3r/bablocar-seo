# Programmatic geo strategy

Документ фиксирует рабочий подход к построению SEO-структуры Bablocar на этапе перехода от очищенной семантики к programmatic SEO.

Главная задача: не придумывать карту сайта из головы, а разложить `semantic/clean_semantic.csv` на управляемые сущности и затем собрать из них гео-матрицу страниц.

## 1. Главный принцип

Структуру строим через последовательную разметку:

```text
очищенная семантика
-> коммерческие интенты
-> алиасы интентов
-> модификаторы
-> правила применимости
-> федеральные хабы
-> гео-страницы
-> страницы с модификаторами
-> статьи / FAQ / инструменты
```

Основная модель для programmatic SEO:

```text
города x коммерческие интенты x разрешенные модификаторы
```

Семантика по всей РФ используется не для точного расчета спроса в каждом городе, а как словарь спроса: какие интенты, алиасы и модификаторы существуют в нише.

## 2. Коммерческий интент

Коммерческий интент — это связка:

```text
тип продукта + объект залога
```

Типы продукта:

- `займ`;
- `кредит`;
- `деньги`;
- `автоломбард`;
- `заложить`.

Жесткое правило:

```text
займ != кредит != деньги
```

Это разные коммерческие интенты, а не алиасы друг друга.

Примеры:

- `займ под залог ПТС`;
- `кредит под залог ПТС`;
- `деньги под залог ПТС`;
- `займ под авто`;
- `кредит под авто`;
- `деньги под авто`;
- `займ под машину`;
- `кредит под машину`;
- `деньги под машину`.

Даже если продуктово Bablocar объясняет услугу через подбор займа у партнеров, SEO-структура может отдельно закрывать спрос со словами `кредит` и `деньги`.

## 3. Объекты залога

Объект залога — это то, под что пользователь хочет получить деньги.

Базовые объекты:

- `ПТС`;
- `авто`;
- `машина`;
- `грузовик`;
- `спецтехника`;
- `мотоцикл`;
- `коммерческий транспорт`.

Правила объединения:

| Формулировки | Объект |
| --- | --- |
| `ПТС` | `ПТС` |
| `авто`, `автомобиль`, `автомобиля` | `авто` |
| `машина`, `машины`, `машину` | `машина` |
| `грузовик`, `грузовое авто`, `грузовой автомобиль` | `грузовик` |
| `спецтехника`, `спецтранспорт` | `спецтехника` |
| `мотоцикл`, `мототехника` | `мотоцикл` |
| `коммерческий транспорт` | `коммерческий транспорт` |

`Авто` и `автомобиль` объединяем.

`Машина` пока держится отдельным объектом, но это рабочая гипотеза. На этапе кластеризации ее нужно проверить по семантике и SERP: если интенты полностью совпадают с `авто`, часть запросов может стать алиасами основного объекта.

## 4. Алиасы

Алиасы — разные формулировки одного и того же интента.

Алиасы не становятся отдельными URL. Они покрываются внутри страницы: текстом, H2, FAQ, анкерами, микроразметкой и внутренней перелинковкой.

Пример:

```text
canonical_intent: займ под залог ПТС
short_name: займ под ПТС
technical_slug: zaim-pod-pts
```

Алиасы:

- `займ под ПТС`;
- `займ по ПТС`;
- `ПТС займ`;
- `взять займ под ПТС`;
- `оформить займ под ПТС`;
- `получить займ под ПТС`.

Пример:

```text
canonical_intent: кредит под ПТС
technical_slug: kredit-pod-pts
```

Алиасы:

- `кредит под ПТС`;
- `кредит под залог ПТС`;
- `кредит по ПТС`;
- `ПТС кредит`.

Пример:

```text
canonical_intent: деньги под ПТС
technical_slug: dengi-pod-pts
```

Алиасы:

- `деньги под ПТС`;
- `деньги под залог ПТС`;
- `деньги по ПТС`;
- `ПТС деньги`;
- `получить деньги под ПТС`.

## 5. Модификаторы

Модификаторы собираются один раз из всей очищенной семантики.

Не нужно собирать отдельный набор модификаторов под каждый интент. Сначала формируется общий справочник, затем через правила применимости решается, к каким интентам модификатор можно применять.

Примеры модификаторов:

- `онлайн`;
- `на карту`;
- `без визита`;
- `без справок`;
- `без подтверждения дохода`;
- `с плохой кредитной историей`;
- `с просрочками`;
- `без кредитной истории`;
- `без стоянки`;
- `без изъятия авто`;
- `авто остается у клиента`;
- `с правом пользования`;
- `срочно`;
- `рефинансирование`;
- `перезалог`;
- `для бизнеса`;
- `для юрлиц`;
- `для ИП`;
- `для физлиц`;
- `для самозанятых`.

Для каждого модификатора задается статус:

| Статус | Значение |
| --- | --- |
| `page` | можно делать отдельную страницу |
| `block` | лучше покрывать блоком на странице |
| `faq` | лучше покрывать FAQ |
| `alias` | алиас другого модификатора |
| `risky_page` | можно только с безопасным шаблоном |
| `risky_no_page` | отдельную страницу не делаем, но спрос не теряем |
| `not_use` | не использовать |
| `check` | нужно проверить вручную |
| `separate_intent` | это не модификатор, а отдельный интент |

Примеры решений:

| Модификатор | Предварительное решение |
| --- | --- |
| `без справок` | `check` / возможно `page` |
| `с плохой кредитной историей` | `check` / возможно `risky_page` |
| `без стоянки` | важный модификатор, проверить как `page` |
| `без изъятия авто` | важный модификатор, проверить как `page` |
| `онлайн` | часто достаточно блока, но можно проверять как страницу |
| `на карту` | `check` |
| `без отказа` | `risky_no_page` по умолчанию |
| `рефинансирование` | возможно `separate_intent` |
| `перезалог` | возможно `separate_intent` |

`Без отказа` и похожие формулировки не теряем из анализа, но не превращаем в публичное обещание. Если такой спрос покрывается, текст должен проходить `partners/safe_claims.md`.

## 6. Правила применимости

Нельзя механически делать:

```text
каждый интент x каждый модификатор
```

Нужна таблица применимости.

Примеры:

| Связка | Решение |
| --- | --- |
| `займ под ПТС` + `без справок` | проверить / возможно можно |
| `кредит под ПТС` + `без справок` | проверить |
| `деньги под ПТС` + `на карту` | проверить |
| `займ под авто` + `без стоянки` | проверить / возможно можно |
| `заложить авто` + `на карту` | скорее нет / проверить |
| любой интент + `без отказа` | `risky_no_page` по умолчанию |

Страница создается, если связка проходит фильтр:

- есть спрос или она нужна для системного покрытия матрицы;
- есть коммерческий смысл;
- можно честно продвигать;
- это не запрещенный или слишком рискованный оффер;
- можно сделать не тонкий дубль;
- есть понятный шаблон страницы.

## 7. URL-модель

Текущую структуру `город -> интент -> модификатор` не ломаем.

Используем:

```text
/{intent}/
/{intent}/{modifier}/
/{geo}/
/{geo}/{intent}/
/{geo}/{intent}/{modifier}/
```

Примеры:

```text
/zaim-pod-pts/
/zaim-pod-pts/bez-spravok/
/samara/
/samara/zaim-pod-pts/
/samara/zaim-pod-pts/bez-spravok/
```

Федеральный уровень нужен как каркас. Основная programmatic-масса — гео-страницы и гео-модификаторы.

## 8. Федеральные хабы

Федеральный хаб — родительская страница коммерческого интента.

Пример:

```text
/zaim-pod-pts/
```

Это полноценный лендинг и структурный центр кластера.

Задачи федерального хаба:

- закрыть общий запрос без города;
- объяснить продукт;
- покрыть алиасы интента;
- дать ссылки на города;
- дать ссылки на соседние интенты;
- дать ссылки на статьи и FAQ;
- принять ссылки от информационного контента;
- помочь поисковику понять структуру сайта.

Федеральные хабы обязательны как каркас, но не являются главной pSEO-массой. Рост строится через гео-сетку.

Пример структуры:

```text
/zaim-pod-pts/
├── первый экран и форма заявки
├── условия
├── документы
├── как работает заявка
├── преимущества
├── FAQ
├── связанные интенты
│   ├── /kredit-pod-pts/
│   ├── /dengi-pod-pts/
│   └── /zaim-pod-avto/
├── популярные города
│   ├── /samara/zaim-pod-pts/
│   ├── /kazan/zaim-pod-pts/
│   └── /moskva/zaim-pod-pts/
└── статьи по теме
```

## 9. Гео-страницы

Гео-страницы — основная масса programmatic SEO.

Пример:

```text
/samara/zaim-pod-pts/
```

Задачи страницы `город + интент`:

- закрыть гео-коммерческий запрос;
- дать локальную релевантность;
- вести в заявку;
- ссылаться на федеральный хаб;
- ссылаться на страницу города;
- ссылаться на соседние интенты в этом городе;
- ссылаться на разрешенные модификаторы.

Пример связей:

```text
/samara/zaim-pod-pts/
├── /zaim-pod-pts/
├── /samara/
├── /samara/kredit-pod-pts/
├── /samara/dengi-pod-pts/
└── /samara/zaim-pod-pts/bez-spravok/
```

## 10. Дерево SEO-архитектуры

Основное дерево строится от федерального интента к гео-страницам этого интента, а затем к разрешенным модификаторам. Ниже пример на одном интенте.

```text
Уровень 1
Главная
/
│
└── Уровень 2
    Федеральный хаб интента
    │
    └── /zaim-pod-pts/
        Канонический интент: займ под залог ПТС
        Короткое название: займ под ПТС
        URL: /zaim-pod-pts/
        Покрывает алиасы:
        - займ под ПТС
        - займ по ПТС
        - ПТС займ
        - взять займ под ПТС
        - оформить займ под ПТС
        - получить займ под ПТС
        │
        └── Уровень 3
            Гео-страницы этого интента
            │
            ├── /moskva/zaim-pod-pts/
            │   Займ под залог ПТС в Москве
            │   │
            │   └── Уровень 4
            │       Модификаторы
            │       ├── /moskva/zaim-pod-pts/bez-spravok/
            │       │   Займ под залог ПТС без справок в Москве
            │       ├── /moskva/zaim-pod-pts/s-plohoi-ki/
            │       │   Займ под залог ПТС с плохой кредитной историей в Москве
            │       ├── /moskva/zaim-pod-pts/bez-stoyanki/
            │       │   Займ под залог ПТС без стоянки в Москве
            │       └── /moskva/zaim-pod-pts/na-kartu/
            │           Займ под залог ПТС на карту в Москве
            │
            ├── /samara/zaim-pod-pts/
            │   Займ под залог ПТС в Самаре
            │
            └── /kazan/zaim-pod-pts/
                Займ под залог ПТС в Казани
```

Правила для этого примера:

- алиас не создает отдельный URL;
- город создает дочернюю гео-страницу интента;
- модификатор может создать дочернюю страницу от `город + интент`, если он разрешен правилами применимости;
- федеральный хаб остается родителем всего интент-кластера.

Городские хабы - не отдельная основная ветка дерева, а поперечная навигация по городу:

```text
/moskva/
└── основные интенты в Москве
    ├── /moskva/zaim-pod-pts/
    ├── /moskva/kredit-pod-pts/
    ├── /moskva/dengi-pod-pts/
    ├── /moskva/zaim-pod-avto/
    └── /moskva/avtolombard/
```

Статьи и FAQ:

```text
/blog/
├── /blog/kak-vzyat-zaym-pod-pts/
├── /blog/dokumenty-dlya-zayma-pod-pts/
├── /blog/chto-budet-esli-ne-platit/
└── /blog/kak-proverit-avto-na-zalog/
```

Статьи ссылаются на федеральные хабы и, где уместно, на выбор города.

## 11. Как обрабатывать очищенную семантику

Работаем сверху вниз:

1. Берем очищенную семантику и сортируем по частотности.
2. Берем самый сильный необработанный коммерческий запрос.
3. Определяем тип продукта.
4. Определяем объект залога.
5. Создаем коммерческий интент.
6. Собираем алиасы этого интента.
7. Отдельно выделяем модификаторы.
8. Отмечаем покрытые запросы как обработанные.
9. Переходим к следующему необработанному коммерческому запросу.
10. Повторяем, пока вся чистая коммерческая семантика не будет покрыта.

Пример:

```text
query: займ под залог ПТС
тип продукта: займ
объект залога: ПТС
интент: займ под залог ПТС
короткое название: займ под ПТС
technical_slug: zaim-pod-pts
```

Алиасы:

- `займ под ПТС`;
- `займ под залог ПТС`;
- `займ по ПТС`;
- `ПТС займ`;
- `взять займ под ПТС`;
- `оформить займ под ПТС`.

Модификаторы:

- `займ под залог ПТС без справок`;
- `займ под залог ПТС с плохой КИ`;
- `займ под залог ПТС без стоянки`.

## 14. Короткие правила

1. Все смысловые сущности в таблицах ведутся на русском.
2. Технический адрес хранится отдельным полем.
3. `Займ`, `кредит` и `деньги` — разные коммерческие интенты.
4. `ПТС`, `авто`, `машина`, `грузовик`, `спецтехника`, `мотоцикл` — объекты залога.
5. `Авто` и `автомобиль` объединяем в объект `авто`.
6. `Машина` пока отдельный объект, но требует проверки по семантике и SERP.
7. Алиасы не становятся URL.
8. Модификаторы собираем один раз из всей семантики.
9. Гео-сетка — основа pSEO-стратегии.
10. Федеральные хабы — обязательный каркас, но не главная масса страниц.
11. Рискованные формулировки не превращаем в обещания.
12. Финальная генерация строится из справочников, а не из отдельных ключей.
