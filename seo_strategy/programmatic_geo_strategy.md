# Programmatic geo strategy

Документ фиксирует рабочий подход к построению SEO-структуры Bablocar на этапе перехода от очищенной семантики к programmatic SEO.

Главная задача: не придумывать карту сайта из головы, а разложить `semantic/clean_semantic.csv` на управляемые сущности и затем собрать из них гео-матрицу страниц.

## 1. Выход этапа

На выходе этапа `SEO-стратегия` должны появиться:

- справочник канонических интентов;
- справочник алиасов интентов;
- справочник модификаторов;
- справочник городов;
- правила применимости модификаторов;
- правила генерации URL;
- гео-матрица pSEO;
- основа для будущей структуры сайта, шаблонов страниц и ТЗ.

Рабочая последовательность:

```text
clean_semantic.csv
-> канонические интенты
-> алиасы интентов
-> модификаторы
-> города
-> правила применимости
-> правила генерации
-> гео-матрица pSEO
-> структура сайта
-> шаблоны страниц
-> ТЗ на страницы
```

Главная модель:

```text
город x коммерческий интент x разрешенный модификатор
```

Семантика по всей РФ используется не для точного расчета спроса в каждом городе, а как словарь спроса: какие интенты, алиасы, модификаторы и рискованные формулировки существуют в нише.

## 2. Смысловые сущности

### Коммерческий интент

Коммерческий интент - это связка:

```text
тип продукта + объект залога
```

Типы продукта:

- `займ`;
- `кредит`;
- `деньги`;
- `автоломбард`;
- `заложить`.

Жесткое правило:

```text
займ != кредит != деньги
```

Это разные коммерческие интенты, а не алиасы друг друга. Даже если продуктово Bablocar объясняет услугу через подбор займа у партнеров, SEO-структура может отдельно закрывать спрос со словами `кредит` и `деньги`.

Примеры канонических интентов:

| Канонический интент | Короткое название | Технический адрес |
| --- | --- | --- |
| `займ под залог ПТС` | `займ под ПТС` | `zaim-pod-pts` |
| `кредит под залог ПТС` | `кредит под ПТС` | `kredit-pod-pts` |
| `деньги под залог ПТС` | `деньги под ПТС` | `dengi-pod-pts` |
| `займ под залог авто` | `займ под авто` | `zaim-pod-avto` |
| `кредит под залог авто` | `кредит под авто` | `kredit-pod-avto` |
| `деньги под залог авто` | `деньги под авто` | `dengi-pod-avto` |
| `автоломбард` | `автоломбард` | `avtolombard` |

### Объект залога

Объект залога - это то, под что пользователь хочет получить деньги.

Базовые объекты:

- `ПТС`;
- `авто`;
- `машина`;
- `грузовик`;
- `спецтехника`;
- `мотоцикл`;
- `коммерческий транспорт`.

Правила объединения:

| Формулировки | Объект |
| --- | --- |
| `ПТС` | `ПТС` |
| `авто`, `автомобиль`, `автомобиля` | `авто` |
| `машина`, `машины`, `машину` | `машина` |
| `грузовик`, `грузовое авто`, `грузовой автомобиль` | `грузовик` |
| `спецтехника`, `спецтранспорт` | `спецтехника` |
| `мотоцикл`, `мототехника` | `мотоцикл` |
| `коммерческий транспорт` | `коммерческий транспорт` |

`Авто` и `автомобиль` объединяем.

`Машина` пока держится отдельным объектом, но это рабочая гипотеза. На этапе кластеризации ее нужно проверить по семантике и SERP: если интенты полностью совпадают с `авто`, часть запросов может стать алиасами основного объекта.

### Алиасы

Алиасы - разные формулировки одного и того же интента.

Алиасы не становятся отдельными URL. Они покрываются внутри страницы: H1/H2, текстом, FAQ, анкерами, микроразметкой и внутренней перелинковкой.

Пример:

```text
канонический интент: займ под залог ПТС
короткое название: займ под ПТС
технический адрес: zaim-pod-pts
```

Алиасы:

- `займ под ПТС`;
- `займ по ПТС`;
- `ПТС займ`;
- `взять займ под ПТС`;
- `оформить займ под ПТС`;
- `получить займ под ПТС`.

Правило: `займ под ПТС`, `займ по ПТС` и `ПТС займ` не создают отдельные URL. Они усиливают страницу `/zaim-pod-pts/` и ее гео-версии.

### Модификаторы

Модификаторы собираются один раз из всей очищенной семантики, а не отдельно под каждый интент.

Примеры модификаторов:

- `онлайн`;
- `на карту`;
- `без визита`;
- `без справок`;
- `без подтверждения дохода`;
- `с плохой кредитной историей`;
- `с просрочками`;
- `без кредитной истории`;
- `без стоянки`;
- `без изъятия авто`;
- `авто остается у клиента`;
- `с правом пользования`;
- `срочно`;
- `рефинансирование`;
- `перезалог`;
- `для бизнеса`;
- `для ИП`;
- `для юрлиц`;
- `для физлиц`;
- `для самозанятых`.

Для каждого модификатора задается статус:

| Статус | Значение |
| --- | --- |
| `page` | можно делать отдельную страницу |
| `block` | лучше покрывать блоком на странице |
| `faq` | лучше покрывать FAQ |
| `alias` | алиас другого модификатора |
| `risky_page` | можно только с безопасным шаблоном |
| `risky_no_page` | отдельную страницу не делаем, но спрос не теряем |
| `not_use` | не использовать |
| `check` | нужно проверить вручную |
| `separate_intent` | это не модификатор, а отдельный интент |

Предварительные решения:

| Модификатор | Рабочее решение |
| --- | --- |
| `без справок` | `check` / возможно `page` |
| `с плохой кредитной историей` | `check` / возможно `risky_page` |
| `без стоянки` | проверить как `page` |
| `без изъятия авто` | проверить как `page` |
| `онлайн` | часто достаточно блока, но можно проверять как страницу |
| `на карту` | `check` |
| `без отказа` | `risky_no_page` по умолчанию |
| `рефинансирование` | возможно `separate_intent` |
| `перезалог` | возможно `separate_intent` |

`Без отказа` и похожие формулировки не превращаются в публичное обещание. Если такой спрос покрывается, текст должен проходить `partners/safe_claims.md`.

## 3. URL-модель

Базовая структура:

```text
/{intent}/                         федеральный хаб интента
/{intent}/{modifier}/              федеральный модификатор, если нужен
/{city}/                           городская страница
/{city}/{intent}/                  основная гео-коммерческая страница
/{city}/{intent}/{modifier}/       гео-страница под модификатор
```

Пример:

```text
/zaim-pod-pts/
/zaim-pod-pts/bez-spravok/
/moskva/
/moskva/zaim-pod-pts/
/moskva/zaim-pod-pts/bez-spravok/
```

Федеральный уровень нужен как каркас. Основная programmatic-масса - гео-страницы и гео-модификаторы.

Не создаем отдельный URL только ради алиаса:

```text
не делаем:
/zaim-po-pts/
/pts-zaim/
/vzyat-zaim-pod-pts/

покрываем внутри:
/zaim-pod-pts/
```

## 4. Глобальная схема на одном интенте

Схема показывает архитектуру сайта на примере интента `займ под залог ПТС`. Это не финальная карта всех URL, а модель, по которой будут строиться остальные интент-кластеры.

```text
/
└── /zaim-pod-pts/                         федеральный хаб интента
    ├── канонический интент: займ под залог ПТС
    ├── короткое название: займ под ПТС
    ├── алиасы на этой же странице:
    │   займ под ПТС, займ по ПТС, ПТС займ,
    │   взять / оформить / получить займ под ПТС
    │
    ├── гео-страницы этого интента
    │   ├── /moskva/zaim-pod-pts/          займ под залог ПТС в Москве
    │   │   ├── /moskva/zaim-pod-pts/bez-spravok/   без справок
    │   │   ├── /moskva/zaim-pod-pts/s-plohoi-ki/    с плохой кредитной историей
    │   │   ├── /moskva/zaim-pod-pts/bez-stoyanki/   без стоянки
    │   │   └── /moskva/zaim-pod-pts/na-kartu/       на карту
    │   │
    │   ├── /samara/zaim-pod-pts/          займ под залог ПТС в Самаре
    │   └── /kazan/zaim-pod-pts/           займ под залог ПТС в Казани
    │
    ├── поперечная навигация по городу
    │   └── /moskva/ -> /moskva/kredit-pod-pts/,
    │                 /moskva/dengi-pod-pts/,
    │                 /moskva/zaim-pod-avto/
    │
    ├── соседние федеральные интенты
    │   ├── /kredit-pod-pts/
    │   ├── /dengi-pod-pts/
    │   └── /zaim-pod-avto/
    │
    └── поддерживающий слой
        ├── статьи / FAQ / инструкции по ПТС
        └── калькуляторы / околоавто-инструменты
```

Главные правила схемы:

- федеральный хаб остается родителем всего интент-кластера;
- алиасы живут внутри страницы, а не плодят URL;
- город создает дочернюю гео-страницу интента;
- модификатор может создать дочернюю страницу от `город + интент`, если он разрешен правилами применимости;
- городские хабы, статьи, FAQ, калькуляторы и инструменты не ломают дерево, а усиливают нужные коммерческие страницы ссылками.

## 5. Что делаем под каждый уровень

### Главная `/`

Роль:

- верхний вход в продукт и SEO-систему;
- объясняет Bablocar как сервис подбора займа под залог ПТС / авто;
- распределяет пользователя по основным направлениям, городам и заявке.

Состав:

- короткое объяснение сервиса;
- основные направления;
- калькулятор / заявка;
- популярные города;
- доверительные и юридические блоки;
- ссылки на ключевые статьи и инструменты.

Линкуется на:

- федеральные хабы интентов;
- популярные городские хабы;
- основные статьи / FAQ / инструкции;
- калькулятор и заявку.

### Федеральный хаб интента `/{intent}/`

Роль:

- родитель всего интент-кластера без города;
- закрывает общий коммерческий запрос;
- объясняет продукт и условия без локальной привязки;
- собирает внутренний вес от статей, FAQ, инструментов и гео-страниц.

Состав:

- H1 под канонический интент;
- первый экран с оффером и CTA;
- калькулятор / заявка;
- условия: сумма, срок, ставка, предварительный расчет;
- документы и требования;
- как работает онлайн-заявка;
- блок про автомобиль у клиента, если релевантно;
- FAQ;
- безопасные уточнения по роли Bablocar и партнерам;
- блок популярных городов;
- блок соседних интентов;
- ссылки на статьи, FAQ и инструменты.

Покрывает:

- канонический интент;
- короткое название;
- алиасы интента;
- часть модификаторов, если они не требуют отдельной страницы.

Линкуется на:

- гео-страницы этого интента;
- соседние федеральные интенты;
- статьи / FAQ / инструкции;
- калькуляторы и инструменты.

### Гео-страница интента `/{city}/{intent}/`

Роль:

- основная коммерческая страница интента в конкретном городе;
- закрывает гео-коммерческий спрос;
- ведет пользователя в заявку;
- связывает федеральный интент с городским контуром.

Состав:

- H1 с городом и каноническим интентом;
- первый экран с заявкой / калькулятором;
- короткое объяснение доступности сервиса в городе без выдуманных локальных фактов;
- условия, документы, требования;
- безопасные уточнения по выдаче, автомобилю, срокам и партнерам;
- блоки под алиасы;
- FAQ;
- ссылки на модификаторы этого интента.

Линкуется на:

- федеральный хаб интента;
- городской хаб;
- соседние интенты в этом городе;
- разрешенные модификаторы;
- релевантные статьи и инструменты.

Нельзя:

- придумывать локальные офисы, партнеров, отзывы или условия;
- делать страницу тонким дублем федерального хаба;
- менять только город без дополнительной полезности.

### Гео-страница модификатора `/{city}/{intent}/{modifier}/`

Роль:

- дочерняя страница под конкретный сценарий внутри города и интента;
- раскрывает не просто ключ, а отдельную пользовательскую задачу;
- помогает покрывать спрос по условиям, способу оформления или проблемной ситуации.

Примеры:

- `/moskva/zaim-pod-pts/bez-spravok/`;
- `/moskva/zaim-pod-pts/s-plohoi-ki/`;
- `/moskva/zaim-pod-pts/bez-stoyanki/`;
- `/moskva/zaim-pod-pts/na-kartu/`.

Состав:

- H1 под город, интент и модификатор;
- быстрый ответ на смысл модификатора;
- калькулятор / заявка;
- уточнение условий без жестких гарантий;
- отличие от основной гео-страницы;
- FAQ именно по модификатору;
- блок безопасных формулировок, если модификатор рискованный.

Линкуется на:

- основную гео-страницу `город + интент`;
- соседние модификаторы;
- федеральный хаб интента;
- релевантную статью или FAQ, если тема требует объяснения.

Страница создается только если:

- есть спрос или системная ценность;
- связка проходит правила применимости;
- модификатор можно честно раскрыть;
- страница не будет тонким дублем;
- формулировки проходят `partners/safe_claims.md`.

### Городской хаб `/{city}/`

Роль:

- поперечная навигация по всем интентам города;
- не является основной вертикалью дерева;
- собирает локальный контур и ведет в гео-коммерческие страницы.

Состав:

- список основных направлений в городе;
- краткое объяснение доступности сервиса;
- ссылки на гео-коммерческие страницы;
- локальные FAQ без выдуманных фактов;
- переходы к федеральным хабам и информационному контенту.

Линкуется на:

- все основные интенты в городе;
- федеральные хабы;
- статьи и инструменты, если это уместно.

### Поддерживающий слой

Роль:

- усиливает коммерческие страницы;
- закрывает информационный и инструментальный спрос;
- помогает безопасно объяснять сложные и рискованные темы.

Состав:

- статьи;
- FAQ;
- инструкции;
- калькуляторы;
- околоавто-инструменты;
- будущие проверки / пробивы.

Линкуется на:

- федеральные хабы;
- релевантные гео-страницы;
- заявку или калькулятор, если это уместно по интенту.

## 6. Перелинковка

Базовая логика:

```text
главная -> федеральные хабы
главная -> популярные города
федеральный хаб -> гео-страницы этого интента
федеральный хаб -> соседние федеральные интенты
городской хаб -> все основные интенты города
гео-страница -> федеральный хаб интента
гео-страница -> городской хаб
гео-страница -> соседние интенты в городе
гео-страница -> разрешенные модификаторы
гео-модификатор -> основная гео-страница
гео-модификатор -> соседние модификаторы
статьи / FAQ / инструменты -> релевантные коммерческие страницы
```

Цель перелинковки:

- показать поисковику родительские и дочерние связи;
- не смешивать алиасы с отдельными страницами;
- передавать вес из информационного слоя в коммерческий;
- не создавать хаотичные петли между всеми страницами подряд.

## 7. Правила применимости

Нельзя механически делать:

```text
каждый интент x каждый город x каждый модификатор
```

Нужна таблица применимости.

Примеры:

| Связка | Решение |
| --- | --- |
| `займ под залог ПТС` + `без справок` | проверить / возможно `page` |
| `кредит под залог ПТС` + `без справок` | проверить |
| `деньги под залог ПТС` + `на карту` | проверить |
| `займ под залог авто` + `без стоянки` | проверить / возможно `page` |
| `заложить авто` + `на карту` | скорее нет / проверить |
| любой интент + `без отказа` | `risky_no_page` по умолчанию |

Страница создается, если:

- есть спрос или системная ценность;
- есть коммерческий смысл;
- можно честно продвигать;
- это не запрещенный или слишком рискованный оффер;
- можно сделать не тонкий дубль;
- есть понятный шаблон страницы;
- есть куда встроить страницу в перелинковку.

## 8. Как обрабатывать очищенную семантику

Работаем сверху вниз:

1. Берем очищенную семантику и сортируем по частотности.
2. Берем самый сильный необработанный коммерческий запрос.
3. Определяем тип продукта.
4. Определяем объект залога.
5. Создаем или находим канонический интент.
6. Собираем алиасы этого интента.
7. Отдельно выделяем модификаторы.
8. Определяем город, если он есть.
9. Проверяем рискованные формулировки.
10. Отмечаем роль запроса в будущей матрице.
11. Переходим к следующему необработанному запросу.

Пример:

```text
query: займ под залог ПТС
тип продукта: займ
объект залога: ПТС
```

## 11. Правила качества

- Не делать отдельный URL только ради одного алиаса.
- Не делать механически все возможные комбинации без правил применимости.
- Не публиковать тонкие дубли, где меняется только город или одно слово.
- Не обещать гарантированную выдачу, отсутствие отказа или одинаковые условия у всех партнеров.
- В гео-страницах не придумывать локальные факты, которых нет.
- В модификаторах раскрывать конкретный смысл страницы, а не повторять основной лендинг.
- Калькулятор и CTA должны быть рядом с коммерческим смыслом страницы.
- Семантику нужно покрывать не только URL, но и структурой контента внутри страниц.
- Финальная генерация строится из справочников, а не из отдельных ключей.
